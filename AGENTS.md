# LMTI Development Constitution

LMTI is the public product name.

LMTI is not a complete Artificial Mind. LMTI is a practical operating memory and verification layer for AI coding agents.

ATLAS is a legacy/internal codename and may still appear in Local Alpha package namespaces or research archives. Project Atlas means the compiled project map generated by `lmti compile`, not a separate product.

Treat `@atlas/*` packages as internal implementation packages of LMTI.

## Agent Boundary

When LMTI is installed inside a host repository, it must be treated as an external intelligence layer, not as a module of the host project.

Connected by context, not by runtime ownership.

Codex must treat LMTI as the project memory/context layer.

Memory is prior belief, not truth. Source code, tests, command output and explicit user instruction remain the source of truth.

Do not bypass privacy gates.

Do not store raw chat, raw secrets, raw customer data, or unverified hallucinations as memory.

## Allowed

- Read LMTI for project context.
- Query policy-safe LMTI memory if available.
- Use LMTI lessons to avoid repeating mistakes.
- Update LMTI only when the task explicitly asks for LMTI changes.

## Forbidden

- Moving LMTI into host application source folders.
- Importing LMTI into host runtime code.
- Refactoring LMTI during normal host product cleanup.
- Deleting LMTI memory/index/lesson files during cleanup.
- Treating LMTI as business logic.

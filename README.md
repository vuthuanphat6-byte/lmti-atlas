# LMTI

Safe project memory for AI coding agents.

LMTI means **Long Memory / Project Intelligence Layer**.

LMTI is a local-first project memory, context routing, privacy, verification, and lesson-capture layer for AI coding agents. It helps agents compile project understanding, retrieve policy-safe context, remember project lessons, and reduce repeated codebase exploration.

Current focus: Codex-first local workflow.

Current scope: Local Alpha. LMTI is not a full AI framework, not a business module of a host application, and not application runtime.

Publish repository: `vuthuanphat6-byte/lmti-atlas`.

## Naming Clarification

LMTI is the public product name.

ATLAS is an internal/legacy codename that may still appear in package namespaces during Local Alpha.

Project Atlas refers to the compiled project map generated by `lmti compile`.

Agents and contributors should treat `@atlas/*` packages as internal implementation packages of LMTI, not as a separate product.

The primary CLI command is `lmti`. If an `atlas` bin is present, treat it as a legacy/internal alias.

The publish repository is `vuthuanphat6-byte/lmti-atlas`. Do not use a private or legacy `/atlas` repository path as the public identity.

## What Is LMTI?

LMTI is a local-first project memory, context routing, privacy, verification, and lesson-capture layer for AI coding agents.

It gives an agent a policy-safe way to retrieve project context, use reviewed memory, and verify stale assumptions before editing.

## Why LMTI?

AI coding agents are fast, but they forget project decisions, permission rules, failed approaches, deployment notes, and privacy constraints. LMTI gives that context a local, reviewable, policy-aware place to live.

## What Problem Does It Solve?

LMTI reduces repeated codebase exploration, stale-memory mistakes, and unsafe context export by routing task-relevant project knowledge through privacy and verification checks.

## What LMTI Does

- Compiles a repository into a Project Atlas / AMF summary.
- Builds policy-safe Context Packs for coding agents.
- Stores structured short-term and long-term project memory.
- Applies privacy gates before context/preflight/adapter output.
- Routes tool execution through a least-privilege SecurityGuard.
- Provides deterministic cognition/world-model helpers for focus and reality checks.

## What LMTI Is Not

LMTI is not:

- A backend module of the host app.
- A frontend module of the host app.
- A business domain module.
- A production app runtime.
- A secret vault.
- A complete AI framework.
- A replacement for source-code verification.
- A model training system.
- A raw chat-history storage system.

## Security Model

LMTI treats project knowledge as sensitive by default.

- No external AI API required.
- No cloud service required.
- No vector database required.
- Secrets are never exported into normal context.
- `.lmti` and `.atlas` local state are ignored and must not be published.
- Target projects are treated as untrusted input during compile.
- Adapter output is egress-scanned and policy-safe by default.
- Privacy gates reduce leakage risk but do not replace dedicated secret scanning.

See `docs/security/` for the threat model, privacy policy, adapter security and release checklist.

## Local Storage

LMTI stores local runtime state under `.lmti/`. This folder may contain project memory, audit logs and local context artifacts. It is intentionally ignored by git.

## Host Project Boundary

LMTI can be installed inside another repository, but it is not part of that repository's application runtime.

LMTI is connected by context, not by runtime ownership.

Agents may read LMTI context and preflight output as guidance. Agents must still verify source code, tests, build output, runtime behavior, and explicit user instructions before editing.

Memory is prior belief, not truth.
